# mini_mal PRD

## Problem Statement

用户面试后想复盘自己的表现，但手动回听录音、整理问答对、思考改进点费时费力。现有方案要么需要手动操作每一步，要么是商业产品无法自部署。

## Solution

一个开源自部署工具：上传或录制面试录音 → 自动转写 → 提取问答对 → 生成改进建议。用户自备 OpenAI Key，通过 Docker Compose 在本机运行，数据不离开自己的电脑。

## User Stories

1. 作为面试者，我想上传面试录音文件，以便系统自动处理
2. 作为面试者，我想在浏览器中直接录制面试，以便当场记录
3. 作为面试者，我想选择录音来源类型（视频会议 / 直接录音），以便系统正确分离说话人
4. 作为面试者，我想查看转写后的文本，以便确认识别是否正确
5. 作为面试者，我想手动编辑转写文本，以便修正术语或识别错误
6. 作为面试者，我想手动标注每个说话人的角色（面试官 / 面试者），以便系统正确提取问答对
7. 作为面试者，我想查看系统从转写中提取的问答对，以便确认提取是否完整
8. 作为面试者，我想对每个问答对查看 1-3 条改进建议，以便针对性地优化回答
9. 作为面试者，我想导出数据为 JSON / CSV / TXT 格式，以便备份或在其他工具中分析
10. 作为面试者，我想删除某条录音及其所有关联数据（转写、问答对、建议），以便管理存储空间
11. 作为面试者，我想在首次使用（API Key 未配置）时收到前端错误提示，以便我知道要去配 Key
12. 作为部署者，我想通过 `docker compose up` 一键启动整个应用，以便快速部署
13. 作为部署者，我想将自备的 OpenAI API Key 填入 .env 文件即生效，以便按自己的用量计费

## Implementation Decisions

### 技术栈

- 后端：Python 3.12 + FastAPI 0.115.x + Uvicorn + SQLAlchemy 2.0.x + SQLite
- 前端：React 19 + TypeScript 5.6.x + Vite 5.4.x + Tailwind CSS 3.4.x
- 容器化：Docker Compose

### 模块划分

测试边界为模块接口（不测实现细节）。

**ASR 转写模块**：音频文件 + 录音来源类型 → 带说话人标签的文本
- 视频会议来源：使用会议软件自带的说话人分离
- 直接录音来源：使用 GPT-4o Transcribe with Diarization

**问答提取模块**：带说话人标签的文本 → 问答对列表
- 依赖角色标注完成（说话人被标记为面试官/面试者）

**建议生成模块**：单个问答对 → 1-3 条改进建议
- 一次性生成，不支持追问

**存储模块**：录音文件（本地文件系统）+ 结构化数据（SQLite）
- 支持删除单条录音及其全部关联数据
- 支持导出 JSON / CSV / TXT 三种格式

### API Key 处理

- 后端启动时不校验 Key，正常启动
- 用户首次触发转写时，若 Key 无效或未配置，后端返回错误，前端弹提示

### 部署

- BYOK：用户自备 OpenAI API Key，填入 .env
- 自部署：Docker Compose 一键启动，全在本机运行
- 不提供在线 Demo 或云端服务

## Testing Decisions

- 每个模块通过接口测试，不测内部实现
- ASR 模块：用已知内容的短音频验证转写输出
- 问答提取：用预置转写文本验证提取数量和内容
- 建议生成：用预置问答对验证建议输出
- 存储：标准 CRUD 接口测试
- 前端：组件级测试 + 关键流程的集成测试

## Out of Scope

- 多用户管理 / 登录
- 在线托管 / 云端部署
- 多维度评分 / 打分系统
- 追问式建议对话
- 移动端原生 App
- 国内 ASR 服务商接入
